  CRITICAL (3):
  ---
    [!!!] Health endpoint extremely slow (34s per call)
         /api/health runs uncached COUNT(*) + GROUP BY source on ALL 517,500 rows every single call.
         Measured: 34s via curl. No caching whatsoever.
         The identical /api/stats endpoint caches for 60s and returns in 1ms on cache hit.
         Fix: Add in-memory cache (30-60s TTL) or reuse /api/stats cached data.
         Code: apps/api/src/index.ts lines 30-48

    [!!!] Listings endpoint catastrophically slow (175-315s per call)
         GET /api/listings triggers COUNT(*) on 500k+ rows for EVERY request with new filters.
         Measured times:
           - Unfiltered default: 175s (first call, no count cache)
           - Filtered (city=Praha): 315s
           - Cached (same unfiltered within 2 min): fast (uses cachedActiveCount)
         The 2-minute count cache in the route handler ONLY works for unfiltered queries.
         Every new filter combination triggers a fresh COUNT(*) that takes 3-5 minutes.
         Root cause: No database-level optimization for COUNT on 500k row table.
         Code: apps/api/src/routes/listings.ts + packages/db/src/queries/listings.ts
         Fix options:
           1. Approximate count: SELECT reltuples FROM pg_class WHERE relname='listings'
           2. Skip count for page > 1 (return total: -1)
           3. Cache filtered counts with longer TTL
           4. Add partial index: CREATE INDEX idx_active ON listings(is_active) WHERE is_active=true
           5. Materialized count view refreshed every few minutes

    [!!!] Server collapses under concurrent load
         100 concurrent requests to /api/stats (a cached endpoint!) resulted in 100% failure.
         All 100 returned connection errors. The single-threaded Node.js server with limited
         PostgreSQL connection pool (idle_timeout: 30s) cannot handle burst traffic.
         After the burst, subsequent requests to /api/health hung for 10+ minutes.
         Fix: Add connection pool sizing, implement request queuing, add rate limiting.

  MAJOR (4):
  ---
    [ !!] CORS allows all origins (wildcard *)
         app.use("*", cors()) in apps/api/src/index.ts line 17 sets no origin restriction.
         Any website can make authenticated API requests.
         Automated test returned Access-Control-Allow-Origin: "undefined" (Hono cors behavior
         when no Origin header value maps to wildcard policy).
         Fix: cors({ origin: ["https://domov.cz", "http://localhost:3000"] })

    [ !!] No rate limiting detected
         No rate limiting middleware configured. Burst of 100 requests crashed the server.
         Fix: Add hono-rate-limiter: 100 req/min for reads, 10 req/min for writes per IP.

    [ !!] No request body size limit
         10MB JSON payload test timed out after 5 minutes (server accepted it but couldn't
         process fast enough). No body-limit middleware prevents large payloads.
         Fix: Add Hono body-limit middleware, max 1MB.

    [ !!] Filtered markers SQL path times out (300s+)
         GET /api/markers?transaction_type=sale&zoom=10&sw_lat=49&sw_lng=12&ne_lat=51&ne_lng=19
         timed out at 300s. The SQL GROUP BY ROUND(lat, precision) query on filtered 500k rows
         is extremely slow. Unfiltered path uses Supercluster (2ms) but filtered falls back to SQL.
         Fix: Consider building filtered Supercluster indexes for common filter combinations,
         or add database indexes optimized for the clustering query.
